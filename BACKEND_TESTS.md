# Тестирование backend-части MultiApp

## Назначение

Тесты проверяют серверную часть FastAPI-приложения без обращения к рабочей базе данных. Для каждого тестового запуска создается временная SQLite-база через переменную `MULTIAPP_DB_PATH`, поэтому файл `multi_app.sqlite` в корне проекта не изменяется.

## Что покрыто

### Авторизация и профиль

- `/api/health` возвращает статус приложения и путь к тестовой базе.
- Защищенные пользовательские эндпоинты требуют авторизации.
- Регистрация нормализует email и создает профиль пользователя.
- Новый пользователь получает пустой аккаунт без демо-данных в финансах, тренировках, ежедневнике и портфолио.
- Профиль позволяет выбрать существующую группу.
- Профиль отклоняет несуществующую группу.
- Валидация регистрации отклоняет некорректные данные.

### Настройки

- Пользователь может сохранить тему, уведомления и напоминание перед парой.
- Некорректная тема отклоняется с ошибкой.

### Финансы

- Создание счета сохраняет тип, баланс и валюту.
- Создание категории и доходной операции обновляет баланс счета.
- Операция с чужим или несуществующим счетом/категорией отклоняется.

### Ежедневник

- Создание категории, задачи, события и заметки проходит через общий CRUD backend.
- Задача переходит в статус выполненной и отражается в статистике ежедневника.
- Событие с окончанием раньше начала отклоняется валидацией.

### Тренировки

- Новый пользователь видит справочники типов тренировок и упражнений, но не получает готовые планы.
- Создание плана, упражнения в плане, записи тренировки и выполненного упражнения сохраняет связанные данные пользователя.
- План с несуществующим типом тренировки отклоняется.

### Администрирование расписания

- Обычный студент не имеет доступа к админскому расписанию.
- Администратор получает сводку, группы и журнал синхронизации.
- Прогресс синхронизации запускается как фоновая задача и возвращает реальный процент выполнения.
- Эндпоинт прогресса доступен только администратору.

## Как запустить

Основной способ запуска — через отдельный Docker Compose сервис `tests`. Он использует временную базу `/tmp/multiapp_test.sqlite`, поэтому рабочий volume приложения не затрагивается.

Запустить все backend-тесты в контейнере:

```powershell
docker compose run --rm tests
```

Команда выводит каждый тест отдельной строкой: путь к файлу, имя теста и статус выполнения. Для успешных тестов будет `PASSED`, для ошибок — `FAILED` или `ERROR`. В конце pytest показывает общий итог и краткую расшифровку проблем.

Запустить один файл:

```powershell
docker compose run --rm tests pytest tests/test_admin_schedule.py
```

Запустить с подробным выводом:

```powershell
docker compose run --rm tests pytest -vv
```

Локальный запуск без Docker тоже доступен. Сначала установить зависимости для разработки:

```powershell
py -m pip install -r requirements-dev.txt
```

Затем запустить все backend-тесты:

```powershell
py -m pytest
```

Запустить один файл:

```powershell
py -m pytest tests/test_admin_schedule.py
```

Запустить с подробным выводом:

```powershell
py -m pytest -vv
```

## Как устроена изоляция

Фикстуры в `tests/conftest.py` перед импортом приложения задают:

```text
MULTIAPP_DB_PATH=<временная папка>/multiapp_test.sqlite
MULTIAPP_TIMEZONE=Asia/Tomsk
```

После этого `server.main` импортируется заново, а база инициализируется уже в тестовом файле. Это позволяет тестировать реальные SQL-миграции, сиды справочников, регистрацию, сессии и API без риска испортить локальные данные.

## Почему парсер расписания не дергает сайт в тестах

В тесте прогресса синхронизации функция массового обновления подменяется на локальную заглушку. Она вызывает `progress_callback` несколько раз и возвращает ожидаемый результат. Так проверяется backend-механика прогресса, не зависящая от доступности сайта ТУСУР и скорости сети.
