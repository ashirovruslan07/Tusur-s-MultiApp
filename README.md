# MultiApp для студентов ТУСУР

MultiApp - веб-приложение для студента ТУСУР, объединяющее учебное расписание, личные финансы, тренировки, ежедневник, портфолио и профиль в одном кабинете.

Приложение рассчитано на две роли:

- студент работает с личным кабинетом, выбирает группу, ведет свои данные и использует модули самоорганизации;
- администратор обслуживает расписание: обновляет факультеты, группы, пары и настраивает автоматическое обновление.

## Основной сценарий работы

1. Пользователь регистрируется или входит в систему.
2. Студент заполняет профиль и выбирает группу: сначала факультет, затем курс, затем конкретную группу.
3. Расписание для выбранной группы загружается из сохраненных данных. Студент не запускает парсинг самостоятельно.
4. Администратор заранее обновляет справочники и расписание всех групп, чтобы у студентов всегда были актуальные данные на текущую и следующую неделю.
5. Остальные модули используют данные пользователя и показывают персональные сводки на главной странице.

## Роли и доступ

### Студент

Студенту доступны:

- главная страница;
- расписание;
- финансы;
- тренировки;
- ежедневник;
- портфолио;
- профиль;
- настройки.

Выбор группы находится только в профиле. Это сделано, чтобы расписание, главная страница и остальные модули использовали один общий источник учебных данных.

### Администратор

Администратору доступен отдельный раздел администрирования расписания. Обычные студенческие модули для администратора скрыты, чтобы интерфейс не смешивал личные функции и управление данными проекта.

Администратор может:

- обновить список групп выбранного факультета;
- обновить группы всех факультетов;
- обновить расписание одной группы;
- массово обновить расписание по факультету, курсу или всем группам;
- настроить автоматическое обновление перед началом следующей учебной недели;
- посмотреть покрытие расписанием, журнал синхронизаций и сводку пользователей по факультетам и группам.

## Функциональные модули

### Авторизация и регистрация

Реализованы регистрация, вход, выход и проверка текущей сессии.

Бизнес-логика:

- email должен быть корректным и уникальным;
- пароль должен быть не короче 6 символов;
- при регистрации пароль подтверждается повторным вводом;
- ФИО должно содержать имя и фамилию;
- после регистрации создаются только профиль и настройки, а личные модули остаются пустыми до действий пользователя;
- сессия хранится в cookie и проверяется при каждом защищенном запросе.

### Профиль

Профиль является главным местом для учебных данных студента.

Реализовано:

- редактирование ФИО, email, телефона, специализации и описания;
- выбор факультета;
- выбор курса после выбора факультета;
- выбор группы после выбора факультета и курса;
- отображение текущей группы, курса и факультета;
- синхронизация выбранной группы с настройками приложения.

Бизнес-логика:

- группа выбирается только в профиле;
- email проверяется на уникальность;
- телефон допускает цифры, пробелы, плюс, дефисы и скобки;
- если студент меняет группу, расписание во всех разделах начинает подгружаться по новой группе.

### Главная страница

Главная страница собирает краткую сводку по всем модулям.

Отображается:

- количество пар сегодня и время первой пары;
- финансовый баланс, доходы и расходы за месяц;
- количество тренировок за последнюю неделю;
- активные задачи и задачи на сегодня;
- ближайшие пары;
- задачи и события;
- последние операции;
- планы тренировок;
- элементы портфолио;
- лента событий на сегодня.

Бизнес-логика:

- данные берутся из всех пользовательских модулей;
- расписание зависит от группы в профиле;
- финансовые показатели считаются по активным счетам и операциям текущего месяца;
- тренировки считаются по журналу выполнения;
- задачи группируются по срокам и статусам.

### Расписание

Модуль расписания показывает пары выбранной группы в удобной недельной сетке.

Реализовано:

- отображение недели по дням и временным слотам;
- карточки пар с дисциплиной, типом занятия, аудиторией и преподавателем;
- визуальное разделение лекций, практик, лабораторных, зачетов и экзаменов;
- отображение текущей группы, факультета и типа недели;
- легенда типов занятий.

Бизнес-логика:

- студент не запускает обновление расписания;
- расписание выбирается по группе из профиля;
- если у группы нет расписания, пользователь видит понятное пустое состояние;
- администрирование расписания вынесено в отдельный раздел администратора.

### Администрирование расписания

Это служебный раздел для подготовки данных расписания.

Реализовано:

- загрузка групп факультета;
- загрузка групп всех факультетов;
- обновление расписания одной группы;
- массовое обновление расписания;
- фильтры по факультету и курсу;
- настройка автоматического обновления;
- ручной запуск автоматического сценария;
- журнал запусков;
- сводка покрытия: сколько групп, сколько расписаний, сколько пар сохранено;
- сводка пользователей по группам и факультетам.

Бизнес-логика:

- расписание обновляется администратором заранее;
- автоматическое обновление запускается перед ближайшим понедельником;
- можно выбрать режим обновления: текущая неделя, следующая неделя или обе недели;
- результаты синхронизации записываются в журнал;
- при повторной загрузке данные недели заменяются актуальной версией.

### Финансы

Модуль финансов помогает студенту вести личный бюджет.

Реализовано:

- счета: карта, наличные, счет, вклад;
- баланс по активным счетам;
- архивирование счетов вместо полного удаления;
- категории доходов и расходов;
- операции дохода и расхода;
- переводы между счетами;
- фильтры по типу операции, счету и поисковой строке;
- аналитика по категориям;
- статистика месяца: доходы, расходы, чистый результат, процент сбережений.

Бизнес-логика:

- операция всегда привязана к счету и категории;
- категория должна соответствовать типу операции: доход или расход;
- расход уменьшает баланс счета, доход увеличивает;
- перевод списывает сумму с одного счета и добавляет на другой;
- перевод на тот же счет запрещен;
- нельзя удалить категорию, если она используется в операциях;
- счет с операциями не удаляется физически, а переносится в архив;
- пользователь видит только свои счета, операции и переводы.

### Тренировки

Модуль тренировок нужен для планирования активности и учета выполненных занятий.

Реализовано:

- справочник типов тренировок;
- справочник упражнений;
- недельные планы тренировок;
- добавление упражнений в план;
- журнал выполненных тренировок;
- запись факта выполнения по конкретному упражнению;
- недельный календарь тренировок;
- статистика: планы, выполненные тренировки, минуты, калории и прогресс.

Бизнес-логика:

- план принадлежит конкретному пользователю;
- упражнение можно добавить только в свой план;
- запись по упражнению можно добавить только в свой журнал;
- день плана ограничен значениями от 1 до 7;
- минуты, калории, подходы, повторы и вес проверяются на корректность;
- при удалении плана удаляется его состав, а старые записи журнала сохраняются без привязки к плану;
- при удалении записи журнала удаляются связанные факты по упражнениям.

### Ежедневник

Ежедневник объединяет задачи, события, заметки и категории.

Реализовано:

- категории ежедневника;
- задачи с приоритетом, статусом и сроком;
- быстрый перевод задачи в выполненные и возврат обратно;
- события с датой, временем, местом и описанием;
- заметки;
- календарь недели;
- блоки задач на сегодня, просроченных, недельных и выполненных;
- фильтры задач по статусу и приоритету.

Бизнес-логика:

- задача должна иметь название;
- приоритет ограничен значениями: низкий, средний, высокий;
- статус ограничен значениями: запланировано, в работе, выполнено;
- дата задачи и события проверяется на корректность;
- окончание события не может быть раньше начала;
- категорию нельзя удалить, если она используется в задачах или событиях;
- пользователь видит только свои задачи, события и заметки.

### Портфолио

Портфолио хранит учебные и личные достижения студента.

Реализовано:

- навыки с уровнем владения;
- проекты;
- достижения;
- сертификаты;
- категории портфолио;
- процент заполненности портфолио;
- последние элементы портфолио на главной странице.

Бизнес-логика:

- проект имеет тип и статус;
- достижение имеет тип, дату, организатора и описание;
- сертификат имеет организацию, дату выдачи, срок действия и ссылку;
- навыки отображаются с уровнем в процентах;
- при удалении проекта, достижения или сертификата удаляются связанные файлы портфолио.

### Настройки

Настройки оставлены только для личных предпочтений пользователя.

Реализовано:

- выбор светлой или темной темы;
- включение или отключение уведомлений;
- настройка напоминания перед парой;
- справочный блок с текущей группой и факультетом;
- быстрые переходы в профиль и расписание.

Бизнес-логика:

- группа не меняется в настройках, а выбирается только в профиле;
- автообновление расписания не настраивается студентом, оно находится в админском разделе;
- напоминание перед парой ограничено диапазоном от 0 до 180 минут.

## База данных

Данные разделены по предметным областям. Ниже описана логическая структура базы.

### Пользователи и доступ

- `Users` - учетные записи пользователей, email, пароль, роль администратора, дата создания.
- `User_Sessions` - активные сессии пользователей.

Связи:

- один пользователь может иметь несколько сессий;
- все личные модули привязаны к `Users.user_id`.

### Учебный профиль и настройки

- `Student_Profile` - ФИО, группа, email, телефон, описание и специализация студента.
- `App_Settings` - пользовательские настройки: тема, уведомления, напоминание, выбранная группа для совместимости с расписанием.

Связи:

- профиль принадлежит одному пользователю;
- профиль связан с группой;
- настройки принадлежат одному пользователю.

### Справочники расписания

- `Faculties` - факультеты, сокращения и коды сайта.
- `Courses` - курсы обучения.
- `Groups` - учебные группы.
- `Schedule_Sync_Settings` - параметры автоматического обновления расписания.
- `Schedule_Sync_Log` - журнал запусков синхронизации.

Связи:

- группа относится к факультету и курсу;
- журнал синхронизации хранит результат каждого массового или ручного обновления.

### Расписание

- `Weekly_Schedule` - неделя расписания для группы.
- `Daily_Schedule` - отдельные пары внутри недели.

Связи:

- одна группа имеет несколько недель расписания;
- одна неделя содержит много пар;
- пара хранит день недели, номер пары, дисциплину, тип занятия, аудиторию, преподавателя и время.

### Финансы

- `Accounts` - счета пользователя.
- `Categories` - категории доходов и расходов.
- `Transactions` - доходные и расходные операции.
- `Transfers` - переводы между счетами.

Связи:

- счет принадлежит пользователю;
- операция принадлежит пользователю, счету и категории;
- перевод принадлежит пользователю и связан с двумя счетами;
- категория может использоваться многими операциями.

### Тренировки

- `Workout_Types` - типы тренировок.
- `Exercises` - упражнения.
- `Workout_Plans` - планы тренировок пользователя.
- `Plan_Exercises` - упражнения внутри плана.
- `Workout_Logs` - выполненные тренировки.
- `Workout_Log_Exercises` - фактическое выполнение упражнений.

Связи:

- план принадлежит пользователю и типу тренировки;
- упражнение в плане связывает план и упражнение;
- запись журнала может быть связана с планом;
- факт выполнения связывает запись журнала и упражнение.

### Ежедневник

- `Planner_Categories` - категории задач и событий.
- `Tasks` - задачи пользователя.
- `Events` - события пользователя.
- `Notes` - заметки пользователя.

Связи:

- задача может быть связана с категорией;
- событие может быть связано с категорией;
- задачи, события и заметки принадлежат пользователю.

### Портфолио

- `Portfolio_Categories` - категории портфолио.
- `Portfolio_Skills` - навыки пользователя.
- `Portfolio_Projects` - проекты.
- `Portfolio_Achievements` - достижения.
- `Portfolio_Certificates` - сертификаты.
- `Portfolio_Files` - файлы, связанные с элементами портфолио.

Связи:

- проекты, достижения, сертификаты и навыки принадлежат пользователю;
- элементы портфолио могут быть связаны с категорией;
- файл может быть связан с проектом, достижением или сертификатом.

## Локальный запуск

Требования:

- Python 3.12 или новее;
- доступ к интернету нужен только для обновления расписания с сайта ТУСУР.

Установка зависимостей:

```powershell
py -m pip install -r requirements.txt
```

Запуск:

```powershell
py -m uvicorn server.main:app --host 127.0.0.1 --port 8000 --reload
```

Открыть приложение:

```text
http://127.0.0.1:8000
```

При первом запуске приложение создает файл данных, выполняет миграции и добавляет стартовые справочники.

## Запуск через Docker

Сборка и запуск:

```powershell
docker compose up --build
```

Открыть приложение:

```text
http://127.0.0.1:8000
```

Данные контейнера сохраняются в volume `multiapp-data`. Это значит, что при перезапуске контейнера пользователи, расписание и записи модулей не пропадут.

Запуск backend-тестов в отдельном контейнере:

```powershell
docker compose run --rm tests
```

По умолчанию тесты запускаются в подробном режиме: в выводе показывается имя каждого теста и его статус `PASSED`, `FAILED`, `SKIPPED` или `ERROR`.

Запуск конкретного файла с тестами:

```powershell
docker compose run --rm tests pytest tests/test_admin_schedule.py
```

Тестовый контейнер использует временную базу `/tmp/multiapp_test.sqlite` и не меняет рабочие данные приложения.

Остановка:

```powershell
docker compose down
```

Остановка с удалением сохраненных данных:

```powershell
docker compose down -v
```

## Тестовые учетные записи

После первого запуска доступны стартовые пользователи:

- студент: `student@example.com`, пароль `student123`;
- администратор: `admin@example.com`, пароль `admin123`.

Эти учетные записи нужны для демонстрации проекта. В рабочем использовании их следует заменить реальными пользователями.

## Проверка работоспособности

Быстрая проверка сервера:

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8000/api/health
```

Проверка Python-кода:

```powershell
py -m py_compile .\server\database.py .\server\main.py
```

Проверка JavaScript:

```powershell
node --check .\public\app.js
```

## Итоговая логика проекта

MultiApp построен вокруг идеи единого студенческого кабинета:

- учебные данные вводятся один раз в профиле;
- расписание готовится администратором и используется студентами без ручного парсинга;
- главная страница объединяет важные события дня;
- финансы, тренировки, ежедневник и портфолио работают как независимые личные модули;
- все пользовательские данные изолированы по владельцу;
- действия в интерфейсе сопровождаются уведомлениями, чтобы пользователь понимал результат операции.
